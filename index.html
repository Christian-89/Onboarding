Urbix Hub Application Flow

This section illustrates the logical sequence of user interactions within Urbix Hub, from initial registration to financial reconciliation.

Each step represents a core process ensuring smooth navigation and system connectivity.

Onboarding (Account Registration & Activation)

The initial process where different user profiles are created and activated.

Manager Registration

  • Business Manager (B2B Client) can register directly.
  • The system validates all input and assigns the user to their organization.
  • Registration is completed after email verification.

Driving school Manager Registration

  • Driving School Manager can register directly.
  • The system validates all input and assigns the user to their organization.
  • Registration is completed after email verification.

Driver Activation

  • Drivers cannot self-register.
  • The Business Manager sends an invitation from their admin dashboard.
  • The driver uses the invite to create their account and link to the organization automatically.

Initial Configuration & Session Package Purchase

Simulator Configuration

The Driving School Manager (DSM) registers available simulators in the system.

  • Includes name, model, cabin type, supported vehicles, address, availability, and image.
  • This setup defines the simulator visibility for Business Managers.

Session Package Purchase

  • The Business Manager purchases session packages via “Plan & Billing.”
  • An invoice with Urbix Hub’s payment details is generated.
  • After payment validation, sessions are credited and ready for booking.

Session Booking & Execution

Both Business Managers and Drivers can manage training session bookings within the platform.

Booking Request (Business Manager)

  • The Business Manager assigns sessions to drivers via “Book A Session.”
  • Required data: driver, city, vehicle type, school, date, and time.

Booking Request (Driver)

  • The Driver can also book a session from their dashboard.
  • Selects city, vehicle type, school, date, and time.

Partner Approval (DSM)

  • The Driving School Manager receives and approves the booking request.
  • The system then notifies all users and confirms the session.

Execution & Attendance Verification

  • The driver attends the session at the driving school.
  • If attended → session marked as Completed and payment is triggered.
  • If absent → session marked as No-Show and subject to refund.

Result Tracking

  • Business Managers can review the session status from their dashboard.
  • Includes reports, performance results, and No-Show incidents.

Financial Reconciliation

Monthly Accumulation

  • The system consolidates all completed and uncompleted sessions monthly.
  • Driving School Managers report session statuses for that month.

School Payments

  • Payments are executed for validated sessions.
  • All reconciliations are logged in the Urbix Hub administrative system.

Welcome to the Urbix Hub Frontend! This section covers everything needed to get up and running with the client application.

Required Tools (Frontend)

  • Node.js (v20 or later)
  • npm or pnpm (preferred package manager)
  • Modern IDE (VSCode recommended)
  • Browser Developer Tools (Chrome/Firefox)

Urbix Hub Frontend Architecture

The frontend is a Single Page Application (SPA) with a focus on user experience.

  • Framework: React + TypeScript
  • Bundler: Vite
  • Styling: Tailwind CSS (utility-first approach)
  • State: Zustand (lightweight state management)
  • Routing: React Router DOM

Frontend Environment & Setup

Frontend configuration variables are managed via the .env file, prefixed with **NEXT_PUBLIC_**:

For this configuration, you are provided with the Environment Variables [FE] file which will help you with the frontend configuration.


                  # Basic Environment
                  NODE_ENV=development
                  PORT=3000
                  # API Configuration
                  NEXT_PUBLIC_API_URL=http://localhost:3001/api
                  NEXT_PUBLIC_GOOGLE_MAPS_API_KEY=YOUR_API_KEY
               

Frontend Execution Flow

               

The Frontend app's execution depends on the API URL configured in the local .env file, which points to the running Backend port.

               

Execution Order (Recommended)

               
                       
  1. Start the Backend: Ensure the backend server is running first to handle API requests.
    2.                    
  2. Start the Frontend: Execute the command below to start the client application.
    3.                
               

Use pnpm as the preferred package manager for a faster and more efficient setup:

                

                  # Navigate to the frontend directory if needed
                  cd urbixhubMVP1_FE
                  
                  # Execute the command to start the development server
                  pnpm run dev
               
               

**Note:** The NEXT_PUBLIC_API_URL in the .env file must match the running backend port (e.g., `http://localhost:3001/api`).

           

Frontend Observability & CI/CD

  • Error Tracking: Sentry for error monitoring.
  • CI/CD: GitHub Actions for testing and building.

Getting Started with Urbix Hub Backend

Before contributing, make sure your environment is properly configured.

Required Tools

  • Node.js (v20 or later)
  • npm or pnpm
  • PostgreSQL
  • TypeORM
  • Docker (optional, for containerized setups)
  • VSCode or any preferred IDE

Project Setup


			git clone https://github.com/urbixhubMVP1_BE.git
			cd urbixhubMVP1_BE

			pnpm install

Ensure your environment variables are configured in the .env file.

Environment & Development Setup

Each developer configures the backend using a local .env file:

For this configuration, you are provided with the Environment Variables file which will help you with the backend configuration.


# Basic Environment
NODE_ENV=development
PORT=3000

# Database
DB_HOST=localhost
DB_USER=postgres
DB_PASSWORD=your_password

# Redis
REDIS_HOST=127.0.0.1
REDIS_PORT=6379

Tip: Connect to Redis and PostgreSQL via secure tunnels for remote access.

 

Database Connection (Remote)

 

The central **PostgreSQL** databases are hosted on **DigitalOcean** for development, staging, and production environments.

 

To connect remotely and interact with the data, use the credentials provided in the team's shared **Environment Variables** document.

   

Required Credentials for Remote Access:

 
       
  • **DB_HOST:** The public connection hostname.
    •    
  • **DB_PORT:** The specific port (usually 25060 or 5432).
    •    
  • **DB_USER:** Your assigned database username.
    •    
  • **DB_PASSWORD:** Your database password.
    •    
  • **DB_NAME:** The specific database name (e.g., urbix_dev).

Backend Execution Flow (Docker)

 

We recommend using Docker to manage the database (PostgreSQL) and caching (Redis) dependencies locally for a faster setup.

 

Required Steps

 
       
  1. Start Docker: Ensure Docker Desktop is running before proceeding.
    2.    
  2. Cleanup (Optional): Stop and remove any previous containers.
    3.    
  3. Launch Containers: Start the database and Redis containers in detached mode.
    4.    
  4. Start Backend App: The application will run on the port specified in your local .env file.
    5.  
 

Execution Commands

  

# Navigate to the backend directory
cd urbixhubMVP1_BE

# 1. Ensure a clean state (optional, but recommended)
docker compose down

# 2. Launch the necessary services (DB, Redis) in the background
docker compose up -d

# 3.The method to retrieve the data from the database is executed.
pnpm run schema:sync

# 4. The application is running
pnpm run dev 
 

Urbix Hub Backend Architecture

The Urbix Hub backend is a distributed Node.js system built for scalability, reliability, and performance.

  • Framework: Node.js + Express (migration to NestJS planned)
  • Database: PostgreSQL with TypeORM entities
  • Cache: Redis for session management
  • Storage: DigitalOcean Spaces for file handling
  • CI/CD: GitHub Actions → Docker → ArgoCD

Observability & CI/CD Overview

  • Logs: Managed via ELK Stack (Elasticsearch, Logstash, Kibana)
  • Deployments: Automated through GitHub Actions & ArgoCD
  • Containers: Dockerized microservices for scalable delivery

All changes are tested, built, and deployed automatically to Kubernetes clusters.

Development Workflow

  • Follow Git branching: main, develop, feature/*, hotfix/*
  • Use semantic commits: feat:, fix:, chore:, etc.
  • Run tests and linting before every push (npm run test)
  • Maintain clean code with ESLint + Prettier
